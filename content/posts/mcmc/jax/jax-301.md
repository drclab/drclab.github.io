Welcome to JAX 301. This post is a conceptual bridge from standard JAX modeling to **vectorized programmable inference**, motivated by Section 1 (Introduction) of *Probabilistic Programming with Vectorized Programmable Inference*.

The core message is simple: modern Bayesian workflows are often bottlenecked by repeated likelihood evaluation, many particles, or many chains. If those repeated computations are conditionally independent, JAX vectorization (`vmap`) can turn them into parallel array programs that run efficiently on accelerators.

## 1. Why this topic now?

Probabilistic programming is increasingly used in settings where inference is the expensive part: simulation-heavy models, large datasets, and custom samplers or proposals. In practice, we often need to scale one (or more) of these dimensions:

- Number of data points evaluated per step.
- Number of particles in Sequential Monte Carlo (SMC).
- Number of chains or proposals in MCMC and variational routines.

In JAX terms, these are usually opportunities to add a **batch axis** and compile once.

## 2. The key idea: programmable inference + vectorization

In many PPL ecosystems, "programmable inference" means we can define custom proposal distributions, kernels, guide programs, or algorithmic control logic. That flexibility is powerful, but it creates pressure on performance.

Vectorization addresses this by expressing repeated work as array transformations:

$$
\text{single-run function } f: \Theta \to Y
\quad \Longrightarrow \quad
\text{batched function } \operatorname{vmap}(f): \Theta^{N} \to Y^{N}.
$$

If the computation graph is compatible with JAX transforms, this can map directly to GPU/TPU-friendly kernels.

## 3. Three practical challenges highlighted by the introduction

Section 1 emphasizes three design constraints that show up immediately in real code.

### 3.1 Compositional vectorization

We do not only want to vectorize model evaluation. We also want to vectorize inference-time logic (for example, many particles or chains). In other words, both model code and inference code should be transformable.

```python
import jax
import jax.numpy as jnp

# Model-level vectorization: evaluate many parameter settings in parallel.
def loglik(theta, x, y):
    logits = x @ theta
    return jnp.sum(y * logits - jnp.logaddexp(0.0, logits))

batched_loglik = jax.vmap(loglik, in_axes=(0, None, None))
```

### 3.2 Vectorized traces

Inference often manipulates rich latent-state objects ("traces"). Under vectorization, we usually want a **struct-of-arrays** representation so each latent field has a leading batch dimension.

```python
trace = {
    "beta": jnp.zeros((num_particles, num_features)),
    "sigma": jnp.ones((num_particles,)),
}
```

This layout is typically easier for JAX transformations than an array of Python objects.

### 3.3 Stochastic branching under `vmap`

Real models branch on random values (mixtures, regimes, adaptive logic). Vectorized execution should preserve semantics while still compiling to efficient array code.

```python
def branch_example(key):
    z = jax.random.bernoulli(key, p=0.3)
    return jnp.where(z, 2.0, -1.0)

keys = jax.random.split(jax.random.PRNGKey(0), 1024)
vals = jax.vmap(branch_example)(keys)
```

## 4. What this means for JAX users

For day-to-day JAX modeling, Section 1 suggests a useful rule of thumb:

1. Write model and inference components as pure functions.
2. Keep latent states in transform-friendly PyTrees.
3. Make batching axes explicit early.
4. Prefer array control flow (`where`, `scan`, `lax.cond`) where possible.

This makes your code more likely to remain composable under `jit`, `grad`, and `vmap`.

## 5. A minimal mental model

You can think of vectorized probabilistic inference as separating concerns:

- **Semantics layer:** model, latent variables, and inference logic.
- **Transformation layer:** `vmap`/`jit`/`scan` over that logic.
- **Execution layer:** optimized array kernels on CPU/GPU/TPU.

When these layers align, custom inference methods stay expressive *and* fast.

## 6. Summary

Section 1 is primarily a motivation and design map:

- Probabilistic programming needs programmable inference for difficult posteriors.
- Programmable inference needs vectorization to scale.
- Vectorization in this setting requires careful handling of traces and branching.

That combination sets the stage for formal details and implementation choices in later sections.
