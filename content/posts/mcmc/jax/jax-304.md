Welcome to JAX 304. This post is based on Section 4 (Implementation) of *Probabilistic Programming with Vectorized Programmable Inference*.

If JAX 303 explains the formal contract, JAX 304 explains the engineering path: how high-level probabilistic code is lowered into JAX/XLA-friendly array programs without losing programmable inference flexibility.

## 1. Big picture: two automation layers

Section 4 frames the compiler as two connected automation layers:

1. **Probabilistic automation**: user-authored `@gen` programs plus inference interfaces like `simulate` and `assess`.
2. **Vectorization automation**: transform those interfaces into vectorized array programs, then lower to XLA.

The implementation strategy is "shallow embedding in Python, deep lowering into JAX primitives."

## 2. Front-end model: `@gen` and named random choices

Users write generative functions in Python and annotate random choices with string addresses. The decorator wraps functions into generative objects exposing interface methods (`simulate`, `assess`, etc.).

```python
# Sketch of user-facing style
@gen
def point(x, a, b, c):
    y_mean = a + b * x + c * x**2
    y = normal(y_mean, 0.2) @ "obs"
    return y
```

This interface-first design is key: inference algorithms compose by calling these methods, not by hard-coding model internals.

## 3. Lightweight effect handlers + partial evaluation

Section 4 describes method implementations via lightweight effect handlers:

- Enter `simulate`: push a handler.
- Encounter `dist(...) @ "name"`: dispatch to handler logic.
- Handler records trace data and computes method-specific outputs.

At first glance this looks Python-heavy. The performance trick is staging with JAX tracing and partial evaluation so Python control/data structures are removed from hot execution paths.

After staging, you get Jaxpr-like IR with array-level primitives only.

## 4. Custom probabilistic primitives in IR

The implementation extends IR with primitives analogous to:

- `sample_p`: probabilistic sampling site.
- `log_dens_p`: primitive density evaluation.

These make transformed programs easier to express and inspect before final lowering.

Later, these probabilistic primitives are eliminated so final compiled code is deterministic array code with explicit PRNG handling.

## 5. Trace representation via PyTrees

Python dictionaries are not directly executable IR values, so traces are registered as PyTrees.

Practical effect:

- Trace objects flatten into nested arrays for transformation.
- JAX transforms (`jit`, `vmap`, etc.) operate on flattened forms.
- Results are reconstructed back into structured traces.

This naturally yields struct-of-arrays behavior under vectorization.

## 6. Vectorizing both inference and models

Section 4 separates two vectorization targets:

1. **Inference methods**: vectorize transformed method Jaxprs directly.
2. **Generative functions themselves**: implement vmapped methods by vectorizing base methods and composing densities appropriately.

The second point matters because it keeps one conceptual path for both modeling and inference batching.

## 7. Stochastic branching with array selectors

For stochastic branching, the implementation relies on array selection semantics (JAX-style `select`) rather than scalar Python branching.

This enables batched conditionals where each element can choose a different branch.

A practical nuance highlighted in Section 4:

- Branch trace schemas may differ.
- Missing branch-specific values can be padded with `NaN` placeholders.
- Selected densities/returns remain valid when combined with selectors.

## 8. Looping with static bounds

Generative analogues of scan-like loops are supported with static loop lengths.

Why static bounds matter:

- XLA compilation requires static shape information.
- Trace shapes must remain compile-time predictable.
- Dynamically sized traces would break shape guarantees.

This is a central constraint when moving from expressive probabilistic semantics to accelerator-compiled execution.

## 9. Final GPU lowering and seeding order

Before GPU execution, custom probabilistic primitives are replaced by explicit PRNG operations.

Section 4 stresses an important ordering rule:

1. Vectorize first.
2. Introduce/split seeds after vectorization.

If you seed too early, batched random sites can accidentally reuse randomness across vector lanes. Seeding after vectorization preserves independence across batched samples.

## 10. Implementation checklist you can reuse

From Section 4, a reusable architecture pattern is:

1. Define high-level probabilistic interfaces.
2. Implement interfaces with lightweight interception logic.
3. Stage early to remove Python overhead.
4. Keep traces transform-friendly (PyTree-compatible).
5. Vectorize method IR, not ad hoc user code paths.
6. Lower probabilistic primitives into explicit PRNG code late.
7. Enforce static-shape constraints where execution traces are materialized.

## 11. Summary

Section 4 shows how to make programmable inference practical on accelerators:

- expressive Python front-end,
- principled interface transforms,
- JAX-native IR staging,
- vectorization-aware trace handling,
- deterministic XLA lowering with correct random seed discipline.

That is the bridge from formal semantics (Section 3) to high-throughput empirical performance (Section 5).
