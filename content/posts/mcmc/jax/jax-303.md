Welcome to JAX 303. This post is based on Section 3 (Formal Model) of *Probabilistic Programming with Vectorized Programmable Inference*.

If JAX 301 motivated vectorized programmable inference and JAX 302 showed the workflow, JAX 303 answers a deeper question: **why should these transformations be trusted?**

Section 3 gives a typed core language and proves that key program transformations preserve probabilistic meaning.

## 1. What the formal model is trying to capture

The paper introduces a core calculus (often written as \(\lambda_{GEN}\)) that mirrors GenJAX-style code:

- Stochastic programs that sample from distributions.
- Traced programs that assign names to random choices.
- Vectorized transformations that batch those programs.

In plain terms, it is a mathematical model of patterns you already write in JAX:

- `simulate`-style sampling,
- `assess`-style density evaluation,
- `vmap`-style batching.

## 2. Type structure: three layers of computation

Section 3 distinguishes three important type-level ideas.

1. `P η`: probabilistic computations.
2. `G_γ η`: traced generative computations.
3. `D η`: density-carrying distributions.

Here, `γ` is a trace "shape" keyed by names of random choices. That grading records not just values, but structure.

A practical interpretation:

- `P η` is "sample some random value of type `η`."
- `G_γ η` is "sample with a structured trace plus return value."
- `D η` is "a distribution object with sampling + density semantics."

## 3. Why traces are typed

Typed traces are a big part of the formal story. A trace is a record keyed by addresses such as `"curve"`, `"ys"`, `"obs"`.

That gives two benefits:

- Compositionality: nested calls create nested trace records.
- Safety: when combining traced blocks, key disjointness and structure are explicit.

This is exactly the behavior we rely on when building programmable inference procedures that inspect or edit portions of latent state.

## 4. Denotational semantics (the trust anchor)

Section 3.2 maps programs to mathematical objects in a probabilistic setting. You can read this as:

- Every well-typed program denotes a concrete probabilistic meaning.
- Generative programs denote both:
  a measure over traces, and
  a return-value function given a trace.

So a generative function is not just "code that runs"; it has a precise interpretation as probability mass/density over structured latent choices.

## 5. `simulate` and `assess` as program transforms

A central point in Section 3.3 is that `simulate` and `assess` are presented as source-to-source transformations.

- `simulate{t}`: run and produce `(trace, return_value, joint_density_at_trace)`.
- `assess{t}`: consume a trace and compute `(return_value, joint_density)`.

The paper states a correctness proposition (Prop. 3.1): both transforms faithfully correspond to the denotational semantics.

Practical meaning for JAX users:

- If your model compiles into these interfaces, density evaluation and simulation are semantically aligned.
- Importance weighting and related methods built from these pieces rest on principled foundations.

## 6. `vmap` as a semantic program transform

Section 3.4 defines vectorization itself as a transform \(vmap_n\{\cdot\}\):

- Types become batched types.
- Contexts become batched contexts.
- Primitive operations are lifted to operate over leading batch axes.

This formalization captures the familiar JAX intuition: batch dimensions propagate systematically through expressions.

A key conceptual result is that vectorization preserves structure while batching values, which explains the array-of-struct to struct-of-arrays effect seen in vectorized traces.

## 7. Why this matters for real implementations

Section 3 is not "theory for theory’s sake." It justifies implementation choices from Section 4:

1. Compile once to trace-aware interfaces.
2. Apply `simulate`/`assess` transformations compositionally.
3. Batch both model and inference code with a single vectorization principle.
4. Expect trace structure to remain stable under batching.

When these hold, you can author custom inference code while still leveraging accelerator-efficient array execution.

## 8. A JAX-oriented checklist from Section 3

When writing your own probabilistic JAX stack, Section 3 suggests this checklist:

1. Keep latent structure explicit and name-addressable.
2. Treat sampling and scoring as first-class transforms.
3. Ensure batched and unbatched interfaces share one semantics.
4. Design trace containers that naturally support struct-of-arrays batching.
5. Compose inference kernels from reusable interface contracts.

## 9. Summary

Section 3 contributes the formal contract behind vectorized programmable inference:

- Typed traced programs (`G_γ η`).
- Mathematically grounded simulation and assessment transforms.
- A vectorization transform that preserves meaning while scaling computation.

That contract is what makes the high-level promises from JAX 301 and JAX 302 credible at scale.
